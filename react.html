JSX & React Fundamentals

The mental model shift: React is a UI function machine. You describe what the UI should look like, React figures out how to get there. JSX is not HTML — it compiles to function calls.

0%
jsx syntax done ✓
JSX & How React Works
JSX is just React.createElement() in disguise. <div className="box"> compiles to createElement("div", {className:"box"}). The browser never sees JSX — Babel transforms it first.
  • JSX compiles to React.createElement() — JSX is not HTML, it's JS
  • JSX rules: className vs class, htmlFor vs for, self-closing tags, one root element
  • Expressions in JSX: {}, ternary, short-circuit — and what you cannot embed
  • React.Fragment and the shorthand <></> — why fragments exist
components done ✓
Components & Props
A component is just a function that returns JSX. Props are read-only — never mutate them. Think of props as function arguments: the parent controls what goes in, the child controls what comes out.
  • Functional components: just a function that returns JSX — naming must be Capitalised
  • Props: passing data down, destructuring props, default prop values
  • children prop: composing components by nesting them like HTML elements
  • PropTypes (or TypeScript) for type-checking props at runtime / compile time
rendering done ✓
Rendering & Lists
The key prop is React's identity card for list items. It must be stable and unique among siblings — never use array index as key if the list can reorder. React uses keys to decide whether to reuse or recreate DOM nodes.
  • Rendering lists with .map() — always returning JSX from the callback
  • The key prop: why it's required, why index-as-key causes bugs
  • Conditional rendering: ternary, &&, early return, switch patterns
  • Rendering null, false, undefined — and what React renders vs ignores
setup done ✓
Project Setup & Tooling
Vite is faster than CRA for development — it serves modules natively over ESM and only bundles for production. CRA is deprecated. For new projects: npm create vite@latest then pick React.
  • Vite + React setup: project structure, index.html, main.jsx, App.jsx
  • React DevTools: component tree, props inspector, profiler tab
  • CSS in React: plain CSS modules, inline styles, classnames library
  • File & folder structure: feature-based vs type-based organisation

Build 8–10 reusable UI components: Button (variants: primary/secondary/danger), Badge, Card, Avatar, Alert, Input, Modal, and Spinner. No hooks yet — props only. Each component accepts props for customisation. This forces you to think in components and nail JSX + props before state exists.

Senior tip: Install React DevTools on day one. Get into the habit of inspecting the component tree, not just the DOM. The DevTools profiler will show you exactly which components re-render and why — a skill you'll use for the rest of your React career.

⬡ Mental Model — Section 1
React components are pure functions: same props in, same UI out. The entire React model is built on this assumption. When React calls your component, it expects no side effects — just return JSX. Side effects (data fetching, subscriptions, timers) belong in hooks, not in the render body.

02

Core Hooks

Hooks are the entire React programming model. useState gives components memory. useEffect synchronises with the outside world. useRef escapes the render cycle. useContext shares state without prop drilling. Master these four and 80% of React clicks.

0%
useState done ✓
useState — Component Memory
State updates are asynchronous and batched. Calling setState schedules a re-render — it does not immediately update the variable. If new state depends on old state, always use the updater function: setCount(c => c + 1).
  • useState signature: const [value, setValue] = useState(initialValue)
  • State updates trigger re-renders — the component function runs again from top
  • Updater function pattern: setState(prev => ...) for state that depends on previous
  • Object and array state: always create new references, never mutate directly
useEffect done ✓
useEffect — Side Effects
useEffect runs after the browser paints. The dependency array controls when it runs: no array = every render, empty array = once on mount, [dep] = when dep changes. Always return a cleanup function to cancel subscriptions, clear timers, or abort fetches.
  • Three forms: no deps (every render), [] (mount only), [dep] (when dep changes)
  • Cleanup function: returned from useEffect, runs before next effect and on unmount
  • Data fetching in useEffect: the async pattern, loading states, error states
  • Strict Mode double-invoke: why effects run twice in dev and what it reveals
useRef done ✓
useRef — Escape Hatch
useRef returns a mutable box: {current: value}. Changing .current does NOT trigger a re-render — that's the entire point. Use it for DOM access, previous values, or any mutable value that shouldn't cause re-renders.
  • DOM refs: ref={myRef} on JSX elements to get the underlying DOM node
  • Mutable values: storing interval IDs, previous state, render counts without re-renders
  • forwardRef: passing refs from parent to child component's DOM node
  • useImperativeHandle: exposing custom methods on a ref (e.g. focus, scrollTo)
useContext done ✓
useContext — Shared State
Context solves prop drilling — passing data through many layers that don't need it. But every consumer re-renders when context value changes. Use context for low-frequency updates (theme, auth, locale). For high-frequency updates (search input), context is the wrong tool.
  • createContext, Provider with value prop, useContext(MyContext) consumer
  • Context with state: combining createContext + useState in a provider component
  • Context re-render problem: all consumers re-render when context value changes
  • Multiple contexts: theme + auth + locale as separate providers stacked at root

⟡

Senior tip: The most common useEffect bug is a missing dependency. Use the eslint-plugin-react-hooks plugin — it will warn you when deps are missing. Never ignore those warnings without fully understanding why. A missing dep means your effect has stale data and will silently produce wrong results.

⬡ Mental Model — Section 2
Every hook call is tied to a specific position in the call order. React identifies hooks by their order — not by name. This is why you can't call hooks inside conditionals: if a hook is skipped, every hook after it shifts position and React reads the wrong state for each one. Rules of hooks are not arbitrary — they're a constraint of the implementation.

Component Patterns

How you structure components determines how maintainable your app is at scale. Controlled forms, lifting state, composition over inheritance, and custom hooks are the four patterns you'll use in every React codebase forever.

0%
forms done ✓
Controlled Forms & Inputs
Controlled inputs: React state is the single source of truth — value={state} + onChange. Uncontrolled inputs: the DOM holds the value, React reads it via a ref. Prefer controlled for forms you validate; uncontrolled for file inputs.
  • Controlled inputs: value + onChange as the only way state flows in React
  • Handling all input types: text, checkbox, radio, select, textarea
  • Form submission: preventDefault, reading state, resetting after submit
  • Basic validation: error state per field, showing/hiding error messages
composition done ✓
Component Composition
Composition is React's answer to inheritance. Instead of extending a base component, you pass components as props or children. The children prop is just a prop that React fills with whatever is between the opening and closing tags.
  • Slot pattern: passing components via named props (header, footer, sidebar props)
  • Render props: passing a function as a prop that returns JSX
  • Compound components: parent + children sharing implicit state (e.g. Tabs/Tab)
  • Higher-Order Components (HOC): wrapping components to inject behaviour
state design done ✓
Lifting State & Data Flow
State lives in the lowest common ancestor that needs it. If two siblings need the same state, lift it to their parent. If that parent is too far away, use context. Data flows down via props; events flow up via callback props.
  • Lifting state: moving state to parent so siblings can share it via props
  • Callback props: child → parent communication via onAction handler props
  • Derived state: compute values from existing state instead of duplicating state
  • Colocation: keep state as close to where it's used as possible — don't over-lift
custom hooks done ✓
Custom Hooks
A custom hook is just a function that starts with use and calls other hooks. The naming convention is what makes React recognise it as a hook. Custom hooks are the primary way to share stateful logic between components.
  • Extracting logic into a custom hook: useFetch, useLocalStorage, useToggle
  • Custom hook return values: returning arrays (like useState) vs objects
  • useDebounce and useThrottle hooks — wrapping vanilla JS utilities in hook form
  • Custom hooks don't share state — each component call gets its own isolated copy

⟡

Senior tip: When you find yourself copy-pasting the same useState + useEffect combination across multiple components, that's the signal to extract a custom hook. Custom hooks are React's answer to utility functions — they make logic portable and testable in isolation.

⬡ Mental Model — Section 3
Think of your component tree like a river. Data flows downstream (props). Events flow upstream (callbacks). The width of the river at each point determines how much state lives there. When two branches need the same water, find the fork point upstream and put the reservoir there — that's lifting state.

04

Advanced Hooks

useReducer replaces complex useState logic with a predictable state machine. useMemo and useCallback are optimisation tools — not defaults. useTransition and useDeferredValue are React 18's concurrency features for keeping UIs responsive under load.

0%
useReducer done ✓
useReducer — State Machines
When your state has multiple sub-values that change together, or when next state depends on complex logic about previous state, useReducer is cleaner than multiple useState calls. Redux was born from this exact pattern.
  • useReducer(reducer, initialState): returns [state, dispatch]
  • Reducer function: pure (state, action) => newState — never mutate state
  • Action objects: { type, payload } convention and exhaustive switch statements
  • useReducer + useContext: the local Redux pattern for complex component trees
memoisation done ✓
useMemo & useCallback
useMemo caches a computed value. useCallback caches a function reference. Both only matter when their output is a dependency of something else (another hook, React.memo). Don't use them everywhere — they add cost.
  • useMemo(fn, deps): memoises the result of an expensive calculation
  • useCallback(fn, deps): memoises the function itself — stable reference across renders
  • React.memo: wraps a component to skip re-render if props haven't changed
  • When NOT to memo: the cost of memoising vs the cost of re-rendering trade-off
layout & id done ✓
useLayoutEffect & useId
useLayoutEffect fires synchronously after DOM mutations but before the browser paints — use it to read/set DOM measurements to avoid visual flicker. useId generates stable server/client-consistent IDs for accessibility attributes.
  • useLayoutEffect vs useEffect: timing difference and when it matters
  • DOM measurements: reading scrollHeight, getBoundingClientRect before paint
  • useId: generating unique IDs for aria-labelledby, htmlFor without hydration mismatches
  • useSyncExternalStore: subscribing to external stores safely in concurrent mode
concurrency done ✓
useTransition & Concurrent React
startTransition marks a state update as non-urgent. React can interrupt it to handle urgent updates (typing, clicking) first. This is the core of React 18 concurrency — keeping the UI responsive while expensive updates happen in the background.
  • useTransition: const [isPending, startTransition] = useTransition()
  • useDeferredValue: defer re-rendering a value until urgent updates complete
  • Suspense: <Suspense fallback={}> boundary for lazy-loaded and async components
  • React 19 use() hook: reading promises and context inside render directly
◆ Practice Project

Build a table of 10,000 rows with live search, sort by column, and pagination. useReducer for all table state (search, sort, page). useMemo for filtered + sorted results. useTransition to keep the search input responsive while filtering renders. React.memo on each row to prevent unnecessary re-renders.
Mark project as complete
⟡

Senior tip: Don't reach for useMemo and useCallback as a first step. Profile first with React DevTools Profiler. If a component is slow, measure before memoising — often the bottleneck is elsewhere. Premature memoisation adds cognitive overhead without measurable benefit.

⬡ Mental Model — Section 4
useReducer is Redux in miniature: one state object, one dispatch function, one pure reducer. The power is that your entire state transitions are in one place and trivially testable — just call reducer(state, action) and assert the result. No React, no DOM, no mocks needed.

React Router v6

Client-side routing makes React apps feel like native apps. React Router v6 is a complete rewrite from v5 — nested routes, layouts, loaders, and actions are first-class features. Understanding the component-based routing model unlocks all of it.

0%
basics done ✓
Router Basics
<Link> renders an anchor tag but intercepts the click — no full page reload. <NavLink> does the same but adds an active class when the URL matches — built for navbars.
  • BrowserRouter + Routes + Route: the three components that set up routing
  • Link and NavLink: client-side navigation without full page reloads
  • useNavigate: programmatic navigation — navigate('/path'), navigate(-1) for back
  • 404 handling: catch-all route with path="*" to render a Not Found component
params & query done ✓
Params, Query & Location
Route params (:id) are part of the path — they identify a resource. Query strings (?sort=asc) are optional filters or UI state. Use params for required resource IDs, query strings for optional filters you want shareable via URL.
  • useParams: reading :id, :slug, :userId from the URL path
  • useSearchParams: reading and writing ?key=value query strings
  • useLocation: reading pathname, search, state, hash — the full location object
  • Passing state via navigation: navigate('/path', { state: { from: 'login' } })
nested routes done ✓
Nested Routes & Layouts
<Outlet> is a placeholder in a parent route component where child routes render. This is how you build shared layouts: a Dashboard layout with sidebar renders once, and child routes (Overview, Settings, Profile) swap in via Outlet.
  • Nested Route config: child routes inside parent Route element
  • Outlet component: where child route components render inside the layout
  • Index routes: the default child rendered at the parent path with no sub-path
  • Layout components: shared shell (header + sidebar + footer) wrapping child routes
advanced done ✓
Protected Routes & Loaders
Protected routes are not a React Router feature — they're a pattern: render the route component only if a condition (auth check) passes, otherwise redirect. The loader API in v6.4+ fetches data before the route renders — similar to getServerSideProps in Next.js.
  • Protected route pattern: wrapper component that checks auth and redirects
  • Navigate component: declarative redirect, replaces useNavigate for JSX redirects
  • Data router (createBrowserRouter): loader, action, useLoaderData, errorElement
  • Lazy routes: lazy(() => import('./Page')) with Suspense for code-split routes

⟡

Senior tip: Synchronise UI state to the URL wherever possible — search terms, filters, active tabs, page numbers. Users should be able to copy the URL and share an identical view. URL-driven state is free persistence and is one of the most underused practices in React apps.

⬡ Mental Model — Section 5
React Router renders components based on URL pattern matching. The entire route tree is just a component tree where the URL is the data. Nested routes render nested components. The Outlet is just a slot where the matched child component drops in. If you understand composition, you understand routing.

State Management

Most apps need three kinds of state: server state (async, cached), client UI state (modals, tabs), and global app state (auth, settings). Each has the right tool. TanStack Query for server state. Zustand for global client state. Context for low-frequency shared state.

0%
context deep dive done ✓
Context API — Production Patterns
Split context into state and dispatch to prevent unnecessary re-renders. Components that only dispatch actions don't need to subscribe to state changes. Two contexts — one for reading, one for writing — is a common production pattern.
  • Context + useReducer: the scalable pattern for complex shared state
  • Splitting state and dispatch into two separate contexts for performance
  • Custom provider components: encapsulating context setup and exposing clean APIs
  • Context limitations: not for high-frequency state — why Zustand / Redux exist
zustand done ✓
Zustand — Simple Global State
Zustand is a minimal store — just a hook. No boilerplate, no Provider needed. Components subscribe to slices of the store; only components that read a changed slice re-render. This makes it far more performant than Context for frequently-updating state.
  • create() store: define state and actions together in one object
  • Selectors: useStore(state => state.count) — subscribe to only what you need
  • Zustand middleware: persist (localStorage), devtools, immer for mutations
  • Slice pattern: organising large stores into separate slice files
server state done ✓
TanStack Query — Server State
Server state is fundamentally different from client state: it's asynchronous, can become stale, needs caching, and can be shared by multiple components. TanStack Query manages all of this automatically — caching, deduplication, background refetch, and stale-while-revalidate.
  • useQuery(queryKey, queryFn): data, isLoading, isError, error, refetch
  • useMutation: POST/PUT/DELETE with onSuccess, onError, optimistic updates
  • Query invalidation: queryClient.invalidateQueries to refetch after mutation
  • Cache config: staleTime, gcTime (cacheTime), refetchOnWindowFocus
redux toolkit done ✓
Redux Toolkit Essentials
Redux Toolkit (RTK) is the official opinionated Redux — no more hand-written action creators. createSlice generates actions and reducer together. RTK uses Immer internally, so you can write mutating syntax that stays immutable under the hood.
  • configureStore and the Redux DevTools integration out of the box
  • createSlice: name, initialState, reducers — auto-generated actions
  • useSelector and useDispatch hooks — reading from and writing to the store
  • RTK Query: a TanStack Query alternative built into Redux Toolkit

Senior tip: 80% of what developers put in Redux is server state — API responses being stored, loading flags being tracked, errors being saved. TanStack Query eliminates all of that. After adopting it, most apps can drop Redux entirely and replace it with Zustand for the remaining client state.

⬡ Mental Model — Section 6
State has three addresses: local (useState inside a component), global client (Zustand store), and server (TanStack Query cache). Before choosing a state tool, ask: where does this data live? If it comes from a server, it belongs in a cache. If it's pure UI behaviour (open/closed, selected tab), it belongs locally or in Zustand.

Performance, Testing & Internals

Understanding how React works under the hood — Fiber, reconciliation, the commit phase — demystifies every performance decision. Testing verifies behaviour, not implementation. This section makes you dangerous in production.

0%
internals done ✓
React Fiber & Reconciliation
Reconciliation is React diffing the new virtual DOM against the previous one to find the minimal set of DOM changes. Fiber is the scheduler underneath — it can pause, prioritise, and resume work. This is what makes concurrent features possible.
  • Virtual DOM: React's in-memory representation of the UI tree
  • Reconciliation diffing algorithm: same type = update, different type = unmount + mount
  • Render phase (pure, can be interrupted) vs commit phase (DOM mutations, side effects)
  • React Fiber: unit-of-work scheduler enabling time-slicing and prioritisation
performance done ✓
Code Splitting & Lazy Loading
Every route you load eagerly adds to the initial bundle. React.lazy + Suspense splits each route into its own chunk — downloaded only when the user navigates there. First load becomes dramatically faster on large apps.
  • React.lazy(() => import('./Component')) + Suspense fallback
  • Route-based code splitting: lazy-loading each page component automatically
  • Error boundaries: class component catching render errors, errorElement in router
  • Bundle analysis: vite-bundle-visualizer to find what's bloating your build
testing done ✓
Testing React Components
Test what the user sees and does, not implementation details. getByRole, getByText, getByLabelText query the DOM as a user would. If your tests break when you refactor but not when user behaviour breaks, you're testing the wrong things.
  • Vitest + React Testing Library: render, screen, userEvent setup
  • Queries: getByRole, getByText, findByText (async), queryByText (optional presence)
  • Testing hooks with renderHook and act() — isolating custom hook logic
  • Mocking fetch / modules: vi.mock, MSW (Mock Service Worker) for API testing
react 19 & beyond done ✓
React 19 & Modern Patterns
Server Components run on the server — no bundle size, direct DB access, zero hydration cost. They cannot use state or effects. Client components are the React you know. Understanding the boundary between them is the core skill for the Next.js / Remix era.
  • React Server Components (RSC): server-only render, no JS shipped to client
  • Server Actions: async functions that run on the server, called from client forms
  • use() hook: reading a Promise or Context inside a component during render
  • useOptimistic: instant UI update before server confirms — the optimistic pattern

⬡ Mental Model — Section 7
React's rendering is split into two phases. The render phase is pure — React calls your components, builds a new fiber tree, diffs it against the old one. No DOM is touched. The commit phase is where DOM mutations, ref assignments, and effects happen. Only the commit phase is synchronous and uninterruptible.
