Week 1

Foundations — Rewire Your Brain

Static typing, Go's type system, pointers, and error handling. Coming from JS, these need the most mental rewiring. Get these right and everything else clicks.

0%
Day 1–2 · ~2 hrs done ✓
Types, Structs & Interfaces
Think: TypeScript interfaces enforced at compile time with zero runtime cost. Structs replace const user = {} — but the shape is locked forever.
  • Static typing vs JS dynamic — thinking in types
  • Structs as compiled JS objects
  • Interfaces — implicit, no implements keyword
  • Pointers — #1 concept Node devs must grok first
Day 3 · ~1 hr done ✓
Error Handling
Every function that can fail returns (result, error). The caller decides what to do. No hidden throws, no surprise crashes.
  • No try/catch — errors are return values
  • The error interface and custom error types
  • Wrapping errors with fmt.Errorf("%w")
  • When to panic vs return an error
Day 4 · ~1 hr done ✓
Collections & Closures
Slices have a length and a capacity — like JS arrays but explicit. Maps are unordered; never depend on iteration order.
  • Slices vs arrays (slices ≈ JS arrays)
  • Maps — key-value like JS objects
  • First-class functions & closures
  • defer — you'll love this once you see it
Day 5 · ~1 hr done ✓
Packages & Modules
Capitalised = exported (public). Lowercase = package-private. The compiler enforces it — no export keyword needed.
  • go.mod vs npm/package.json
  • Exported vs unexported (Capital = public)
  • Stdlib: fmt, os, strings, io
  • go get to add dependencies
◆ Weekend Project
CLI Task Manager
Build a command-line todo app with cobra. Persist tasks to a JSON file using encoding/json. Forces you to use structs, file I/O, error handling, and package structure all at once — the perfect first real Go project.
Mark project as complete
⟡

Senior tip: Go's biggest shift isn't the syntax — it's the zero value philosophy and nil vs undefined. Read Effective Go on initialization before the weekend project. Also set up gopls in your editor; the LSP feedback is your best teacher.

⬡ Mental Model — Week 1
Go is not "JavaScript with types". It's a compiled systems language that chose simplicity over features. You lose: classes, inheritance, generics complexity, exceptions. You gain: fast compile, predictable memory, zero-cost abstractions, and code that reads like prose. The constraint is the feature.
Types & structs Error handling Interfaces Go modules Pointers defer
Week 2

HTTP, Routing & Databases

Your home turf — you've built REST APIs in Express. Rebuild that intuition in Go. Your Node.js patterns map almost 1-to-1, but now they're faster and statically typed.

0%
Day 1 · ~1 hr done ✓
net/http Raw Server
Go's stdlib HTTP is production-grade without a framework. Understanding it makes Gin feel like sugar on top — not magic.
  • HTTP handlers — like Express route callbacks
  • ResponseWriter + *http.Request
  • ServeMux for basic routing
  • Middleware pattern via function wrapping
Day 2 · ~1 hr done ✓
Gin Framework
gin.Context is your req + res in one. ShouldBindJSON(&input) replaces req.body with compile-time type safety.
  • Gin is the Express of Go — very familiar feel
  • Route groups, path params, query strings
  • Middleware: auth, logging, CORS
  • JSON binding & validation via struct tags
Day 3 · ~1 hr done ✓
PostgreSQL with sqlx
No ODM magic. You write SQL, sqlx scans rows into structs. Explicit beats magic — you see exactly what hits the DB.
  • sqlx: raw SQL with struct scanning
  • Prepared statements & named queries
  • Transactions — critical for data integrity
  • Connection pooling config (vs Mongoose)
Day 4–5 · ~2 hrs done ✓
Project Structure & Config
cmd/ = entrypoint. internal/ = app logic (import-protected by compiler). pkg/ = reusable across projects.
  • Standard layout: cmd/, internal/, pkg/
  • Env config with viper or godotenv
  • JWT auth middleware in Go
  • Repository pattern — Mongoose models reimagined
◆ Weekend Project
Notes App REST API with Auth
Full CRUD REST API — Gin + PostgreSQL + JWT auth. Focus on proper layering: handlers → services → repository. Same architecture you'd build in Node, but now statically typed and 3x faster.
Mark project as complete
⟡

Senior tip: Don't jump to GORM yet. Learn sqlx with raw SQL first — GORM hides too many gotchas that will bite you in production. Also learn context.Context for request timeouts early. There's nothing like it in Express, and it's everywhere in production Go code.

⬡ Mental Model — Week 2
Gin is Express minus the callback hell. The key shift: request handling is synchronous Go code — no async/await needed because each request runs in its own goroutine. context.Context is the request-scoped value bag that flows through every layer, carrying deadlines, cancellation, and request IDs. Learn it early.

Week 3

Concurrency — Go's Superpower

This is why people choose Go over Node for backend services. Goroutines aren't threads and channels aren't event emitters — but once it clicks, it feels more powerful than async/await.

0%
Day 1 · ~1 hr done ✓
Goroutines & Channels
go func() costs ~2KB stack vs ~1MB for OS threads. Run 100K goroutines easily. Channels are the typed pipes between them.
  • Goroutines: lighter than threads, ~2KB stack
  • Channels: typed pipes between goroutines
  • Buffered vs unbuffered channels
  • select — like Promise.race but for channels
Day 2 · ~1 hr done ✓
sync Package & WaitGroups
Use channels to transfer data ownership. Reach for Mutex only when multiple goroutines must share state they all own.
  • WaitGroup — like Promise.all for goroutines
  • Mutex for shared state (prefer channels)
  • sync.Once for singleton initialization
  • Race detector: go run -race
Day 3 · ~1 hr done ✓
Worker Pools
Classic pattern: N workers, one jobs channel, one results channel. Workers pull jobs, push results. Simple, composable, no framework needed.
  • Fan-out / fan-in — Go's most powerful pattern
  • Fixed-size worker pool with channels
  • Graceful shutdown with context.WithCancel
  • Real use: parallel DB queries, batch jobs
Day 4–5 · ~2 hrs done ✓
Testing in Go
_test.go files are compiled out of production binaries automatically. Table-driven tests = one function, N scenarios as a struct slice.
  • Built-in testing package — no Jest needed
  • Table-driven tests — Go's idiomatic style
  • Mocking with interfaces (no jest.mock)
  • Benchmarks with testing.B
◆ Weekend Project
Concurrent URL Health Checker
Check 100+ URLs concurrently using a worker pool (10 workers). Return health status, response times, HTTP codes. Add it as an endpoint to your Week 2 API. Write table-driven tests for the core logic. Exercises goroutines, channels, WaitGroups, and testing all at once.
Mark project as complete
⟡

Senior tip: "Don't communicate by sharing memory; share memory by communicating." When you reach for a Mutex, ask if a channel would express the intent more clearly. Always run concurrent code with -race during development — it catches bugs that only appear under production load.

⬡ Mental Model — Week 3
Node's event loop handles concurrency on one thread. Go's scheduler multiplexes thousands of goroutines across all CPU cores. When a goroutine blocks on I/O, the scheduler parks it and runs another — no callbacks, no promises, no async tax on the call stack. This is why Go HTTP servers handle 10× more concurrent requests per memory than Node.
Goroutines Channels Worker pools sync package Table-driven tests
Week 4

Production Go — Ship It

The week that turns you from someone who can write Go into someone who can run Go in production. Observability, profiling, Docker, and a capstone you can show in interviews.

0%
Day 1 · ~1 hr done ✓
Logging & Observability
JSON logs + structured fields = grep-able, Datadog/Loki-indexable. console.log strings are unsearchable under load.
  • Structured logging with slog (Go 1.21+) or zap
  • Request tracing via context propagation
  • Prometheus metrics with promhttp
  • Liveness vs readiness health endpoints
Day 2 · ~1 hr done ✓
Performance & Profiling
go tool pprof shows where CPU time actually goes. Most perf issues are GC pressure or a hot allocation loop — measure before optimising.
  • pprof — CPU & memory profiling built in
  • Escape analysis: heap vs stack allocation
  • Avoiding goroutine leaks in production
  • Benchmark-driven optimization loop
Day 3 · ~1 hr done ✓
Docker & Deployment
Stage 1: golang:alpine builds the binary. Stage 2: scratch (empty image) just runs it. Result: <10MB, no shell, no CVEs.
  • Multi-stage Dockerfile — single binary output
  • <10MB container with scratch or alpine
  • Graceful shutdown: os.Signal + Shutdown()
  • 12-factor config via environment variables
Day 4–5 · ~2 hrs done ✓
gRPC Basics
HTTP/2 + binary protobuf = 5–10× faster than JSON REST for internal services. Schema-first means client and server agree upfront — no more API drift.
  • Why gRPC for service-to-service comms
  • Protobuf: define your API contract first
  • Generating Go code from .proto files
  • Unary RPC — one simple server + client
◆ Capstone Project
Production-Ready URL Shortener
Everything together: Gin REST API + PostgreSQL + Redis caching + background goroutine for click analytics + structured logging + Prometheus metrics + multi-stage Docker build. Under 10MB container. Deploy to Railway or Fly.io. This is the portfolio piece for backend interviews.
Mark capstone as complete
⟡

Senior tip: Go's single binary is its killer Docker feature. A Node app needs ~300MB of node_modules in the image; your Go binary is often under 20MB total. That difference matters at scale — faster cold starts, smaller attack surface, cheaper egress. Always implement http.Server.Shutdown with a signal handler in every production service.

⬡ Mental Model — Week 4
Production Go means: one statically-linked binary, structured JSON logs, metrics that Prometheus can scrape, a /healthz endpoint Kubernetes can probe, and a signal handler that drains in-flight requests before exit. This is the entire operational surface. No runtime, no VM, no dependency hell — just a binary and env vars.
