Week 1

JavaScript & CSS — The Interview Layer

You know JS. Now you need to understand why it works. Closures, the event loop, prototype chain, async patterns — interviewers test these exact concepts, not whether you can use React.

0%
Day 1–2 · ~4 hrs done ✓
JS Engine & Scope
Interviewers love asking "what happens here?" — knowing how JS parses & executes code is your answer to 90% of tricky questions.
  • var / let / const — hoisting & temporal dead zone
  • Closures — lexical scope, practical use cases
  • this — 4 binding rules (default, implicit, explicit, new)
  • Prototype chain & class as syntax sugar
Day 3 · ~2 hrs done ✓
Async JavaScript — Full Picture
The event loop is the #1 JS interview topic. Know the difference between microtasks and macrotasks — it explains every "why does this run first?" question.
  • Event loop: call stack, task queue, microtask queue
  • Promises — .then() chaining, error propagation
  • async/await — why it's just promise sugar
  • Promise.all / allSettled / race
Day 4 · ~2 hrs done ✓
Modern JS Patterns
These are the patterns you'll write every day on the job. Destructuring, spread/rest, and array methods separate "knows JS" from "thinks in JS".
  • Destructuring — objects, arrays, default values, rename
  • Spread / rest operators — shallow copy gotchas
  • map / filter / reduce — functional array style
  • Optional chaining ?., nullish coalescing ??
Day 5 · ~2 hrs done ✓
CSS That Actually Matters
Interviewers don't test divs and colours — they test layout. "Build this UI in 15 mins" is a common screen. Know Flexbox and Grid cold.
  • Flexbox mastery — gap, align-items, justify-content
  • CSS Grid — template areas, auto-fill, minmax
  • CSS variables + Tailwind utility class mindset
  • Responsive design — mobile-first, clamp(), breakpoints
◆ Weekend Project
JS Interview Prep Kit
Build a personal "JS concepts" repo on GitHub. For each topic: write a 10-line code snippet that demonstrates the concept + a comment explaining what would trip people up. Implement debounce, throttle, memoize, and a Promise.all polyfill from scratch. Interviewers love "implement X without a library".
Mark project as complete
🎯

Interview hot topics this week: "What is a closure?" (answer with a real use case, not just a definition). "What's the difference between == and ===?" "What happens in the event loop when you call setTimeout(fn, 0)?" These come up in nearly every JS screen.

⟡

Senior tip: When asked about closures, don't just say "a function that has access to its outer scope." Show the interviewer you understand why — mention the lexical environment, the fact the inner function holds a reference (not a copy) to the outer variable, and give a real use case like a counter factory or a debounce function.

⬡ Mental Model — Week 1
JavaScript is single-threaded, but it's not slow — it delegates I/O to the browser APIs / Node APIs, and uses the event loop to pick up results without blocking. Every async interview question is really about this: "which queue does this go in, and when does the engine drain it?" Know that Promise.then → microtask queue (runs before the next task), setTimeout → macrotask queue (runs after). That's the whole model.

Week 2

React — From User to Architect

You've used React. Now understand what's happening under the hood. Reconciliation, the rules of hooks, performance pitfalls — this is what separates junior from mid-level in interviews.

0%
Day 1 · ~2 hrs done ✓
React Internals
You don't need to read React source — but knowing what reconciliation means tells you why keys matter, why mutating state breaks things, and why re-renders happen.
  • Virtual DOM — what it is and why it's fast
  • Reconciliation & diffing — how React decides what to update
  • The key prop — why it matters beyond lists
  • Rules of hooks — why you can't call in conditions
Day 2 · ~2 hrs done ✓
Hooks — Deep & Practical
Don't memorise the API — understand when and why each hook exists. "When would you use useCallback?" is a very common interview question.
  • useState batching — React 18 auto-batching behaviour
  • useEffect — deps array, cleanup, common mistakes
  • useCallback & useMemo — when they actually help
  • useRef — DOM refs vs mutable instance values
Day 3 · ~2 hrs done ✓
Custom Hooks & State Patterns
Custom hooks are the "how do you structure reusable React logic?" answer. useReducer is the answer to "how do you manage complex state without Redux?"
  • Custom hooks — useFetch, useLocalStorage, useDebounce
  • useReducer — when state transitions get complex
  • Context API — global state, provider pattern
  • State colocation — lifting up vs pushing down
Day 4–5 · ~4 hrs done ✓
React Router & Performance
Route-based code splitting drops initial bundle size by 40–60% in most apps. Knowing this is a huge green flag in interviews about "how would you optimise this?"
  • React Router v6 — nested routes, Outlet, layout routes
  • Protected routes — auth guard pattern
  • Code splitting with React.lazy + Suspense
  • React.memo — prevent re-renders, measure first
◆ Weekend Project
React SPA with Auth Flow
A multi-page React app with: protected routes (redirect if not logged in), a custom useAuth hook, a custom useFetch hook with loading/error states, at least one page using useReducer for form state, and React Router v6 nested layouts. Use a free public API for data. This is your portfolio React project.
Mark project as complete
🎯

Interview hot topics this week: "Why did my component re-render?" "Explain the difference between useCallback and useMemo." "How does the virtual DOM work?" "How do you share state between sibling components?" These are the questions that trip up people who've only used React without understanding it.

⟡

Senior tip: Most juniors overuse useMemo and useCallback. The real answer in interviews: "I only add them after measuring a performance issue with React DevTools Profiler." Premature memoisation hurts performance. Saying this in an interview signals senior thinking.

⬡ Mental Model — Week 2
React's job is to keep the UI in sync with state. Every re-render is React asking "what should the DOM look like now?" The reconciler diffs the new element tree against the previous one and applies only the changes. Hooks are the mechanism to "hook into" React's render cycle from a function component — that's why their call order must be stable between renders. Break that order, break the mapping.

Week 3

Node, Express & MongoDB — Backend Like a Pro

You've built Express routes. Now go deeper — middleware chains, proper error handling, JWT refresh flow, Mongoose schema design. This is where backend interviews separate candidates.

0%
Day 1 · ~2 hrs done ✓
Node.js Runtime Depth
Node's event loop is different from the browser's. Understanding libuv, the thread pool, and the timer phases will answer questions like "is Node single-threaded?"
  • Node event loop phases — timers, poll, check, callbacks
  • Streams & Buffers — why you don't readFileSync in production
  • CommonJS vs ES Modules — require vs import
  • Environment config — dotenv, process.env, never commit secrets
Day 2 · ~2 hrs done ✓
Express Architecture
Middleware is Express. Every route handler, every error handler, every auth check — it's all middleware. Understanding the chain means you can debug any Express issue.
  • Middleware chain — order matters, next() flow
  • Centralised error handling — next(err) pattern
  • Request validation — Joi or express-validator
  • Layered architecture: routes → controllers → services
Day 3 · ~2 hrs done ✓
Auth — Production Grade
JWT auth is in every MERN job description. Go beyond "sign and verify" — understand the access + refresh token pattern, why httpOnly cookies beat localStorage for security.
  • bcrypt hashing — salt rounds, timing attack prevention
  • JWT — access token (short-lived) + refresh token (long-lived)
  • httpOnly cookies vs localStorage — XSS trade-offs
  • Role-based access (RBAC) middleware pattern
Day 4–5 · ~4 hrs done ✓
MongoDB & Mongoose Mastery
"Embedding vs referencing" is the MongoDB schema interview question. Knowing when to use aggregation vs populate() signals you've built something bigger than a tutorial app.
  • Schema design — embedding vs referencing (denorm vs norm)
  • Indexes — compound, text, sparse, explain query plan
  • Mongoose middleware (pre/post hooks) and virtuals
  • Aggregation pipeline — $match, $group, $lookup
◆ Weekend Project
REST API with Layered Architecture
Build a Node/Express API with proper separation: routes/ → controllers/ → services/ → models/. Include: JWT auth with refresh tokens, role-based middleware (admin/user), request validation, centralised error handler, and at least one aggregation query. This is the backend you'll reference in every interview.
Mark project as complete
🎯

Interview hot topics this week: "What is middleware in Express?" "How does JWT authentication work?" "What's the difference between embedding and referencing in MongoDB?" "How would you handle errors globally in Express?" These are word-for-word questions from real MERN interviews.

⟡

Senior tip: Never store JWTs in localStorage in production — XSS can steal them. Use httpOnly cookies for the refresh token and keep the access token in memory (a JS variable). Most candidates don't know this distinction, and knowing it will impress every interviewer asking about authentication security.

⬡ Mental Model — Week 3
An Express app is a list of middleware functions. Every request walks down the list until something calls res.send() or the list ends. Controllers shouldn't know about the DB — that's the service's job. Services shouldn't know about HTTP — that's the controller's job. This separation means you can test each layer in isolation, and swap MongoDB for Postgres without touching your routes.

Week 4

Full Stack Integration & Interview Ready

Connect everything, secure it, test it, deploy it, and walk into interviews with a working production app and confident answers to every MERN question they can throw at you.

0%
Day 1 · ~2 hrs done ✓
Security & Production Patterns
Every production MERN app needs these. "How do you secure an Express API?" is a standard interview question — and this is the complete answer.
  • CORS — origin whitelist, credentials, preflight
  • helmet.js — HTTP security headers in one line
  • Rate limiting — prevent brute force on auth endpoints
  • Input sanitisation — XSS & NoSQL injection prevention
Day 2 · ~2 hrs done ✓
Testing Your Stack
You don't need 100% coverage. Know how to write a meaningful test for a React component and an Express endpoint — that's what employers actually ask about.
  • Jest unit tests — pure functions, utilities, transformations
  • React Testing Library — test by user behaviour, not implementation
  • Supertest — HTTP API endpoint testing without a running server
  • What to test: happy path + one edge case per feature
Day 3 · ~2 hrs done ✓
Deploy & Go Live
A live link beats a GitHub repo every time. "It's deployed at…" is the most powerful phrase in a job interview. Get both frontend and backend live before you start applying.
  • Vercel for React — vercel.json, env vars, preview deploys
  • Railway or Render for Node API — free tier, env config
  • MongoDB Atlas — cloud cluster, IP whitelist, connection string
  • GitHub Actions — auto-lint + test on every push to main
Day 4–5 · ~4 hrs done ✓
Interview Blitz Prep
Interviews are a skill, not just knowledge. Practice saying your answers out loud. The candidate who can explain a concept clearly always beats the one who vaguely "knows" it.
  • Top 20 MERN interview questions — written answers for each
  • REST API design — status codes, pagination, versioning
  • Portfolio README — live link, tech stack, architecture diagram
  • System design basics — how would you build Twitter/Airbnb frontend?
◆ Capstone Project
Full Stack MERN — Portfolio Centrepiece
Combine your Week 2 React SPA + Week 3 Node API into one deployed, production-quality app. Requirements: user auth (register/login/logout), at least 3 CRUD features, protected routes on the frontend, RBAC on the backend, rate limiting, input validation, deployed live with working demo. Suggested ideas: a job tracker, a recipe book with sharing, a note-taking app with tags. This is what you demo in every interview.
Mark capstone as complete 🎉
🎯

Interview hot topics this week: "Walk me through your project architecture." "How do you handle security in your app?" "How would you improve the performance of your React app?" "What would you do differently if you had to rebuild it?" Having a live deployed app lets you answer all of these with confidence and specifics.

⟡

Senior tip: Before any interview, open your deployed app and be ready to walk through the code live. Interviewers will often say "show me something you built." Walk them through one interesting technical decision — why you chose to embed vs reference in Mongo, or why you use httpOnly cookies. One specific, thoughtful decision tells them more than listing ten technologies.

⬡ Mental Model — Week 4
A production MERN app is four separate concerns that communicate over HTTP: a React SPA (client-side rendering, routing, state), an Express API (business logic, auth, validation), a MongoDB Atlas cluster (persistent data), and a deployment platform (env vars, TLS, uptime). Interviewers care that you can reason about the seams — CORS, auth token flow, data serialisation, error handling across the boundary. That understanding is what the interview is testing.
