01

HTML & CSS Foundations

Every web interface starts here. HTML gives structure and meaning. CSS gives style and layout. These aren't just prerequisites — they're skills you'll use daily alongside JavaScript.

0%
structure done ✓
HTML Structure & Semantics
Use semantic elements everywhere — <article>, <nav>, <main>. A <div> has no meaning; a <header> tells browsers, screen readers, and search engines exactly what it is.
  • Document structure: DOCTYPE, html, head, body, meta charset & viewport
  • Semantic elements: header, nav, main, section, article, footer, aside
  • Forms & inputs: form, input types, label, textarea, select, button
  • Accessibility: alt text, tabindex, ARIA labels and landmark roles
core styling done ✓
CSS Core Concepts
The box model is the foundation of all CSS layout. Every element is a box — content, padding, border, margin. Add box-sizing: border-box globally and never fight CSS again.
  • Selectors: element, .class, #id, attribute, :hover, :focus pseudo-classes
  • The box model: content, padding, border, margin, box-sizing: border-box
  • Display values: block, inline, inline-block, none — and how they affect layout
  • Specificity & cascade: how browsers resolve conflicting CSS rules
layout done ✓
Flexbox, Grid & Responsive
Flexbox is 1D (row or column). Grid is 2D (rows AND columns). Use Flexbox for components, Grid for page structure. Media queries are mobile-first: min-width, not max-width.
  • Flexbox: flex-direction, justify-content, align-items, flex-wrap, gap
  • CSS Grid: grid-template-columns, grid-area, fr unit, auto-fill / auto-fit
  • Positioning: relative, absolute, fixed, sticky — and z-index stacking
  • Responsive design: mobile-first approach, @media queries, viewport meta
modern css done ✓
Modern CSS Features
CSS custom properties are not just convenience — every major design system (Tailwind, Material UI, shadcn) is built on them. Master them now; they're everywhere.
  • CSS custom properties: --var-name, var(), :root scope, theming with variables
  • Transitions & animations: transition shorthand, @keyframes, animation properties
  • Pseudo-elements: ::before, ::after — the content property and use cases
  • CSS units deep dive: px, rem, em, %, vw/vh — and clamp() for fluid sizing
◆ Practice Project
Responsive Portfolio Page
Build a personal portfolio with HTML only — no JS yet. Sections: hero, about, projects grid, contact form. Use semantic HTML throughout. Style it with Flexbox + Grid, CSS variables for colours, smooth transitions on hover, and make it fully mobile responsive.
Mark project as complete
⟡

Senior tip: Don't rush past HTML semantics — they directly affect SEO and accessibility. Use the browser's accessibility inspector (DevTools → Accessibility tab) to verify your semantic structure reads correctly. And never use <div> when a semantic element exists.

⬡ Mental Model — Section 1
HTML is the skeleton, CSS is the skin, JavaScript is the muscles. You can't write good JS DOM manipulation code if you don't deeply understand what the DOM tree looks like. Every hour spent on HTML/CSS saves three hours debugging layout issues from JS.

02

JavaScript Fundamentals

The core language — types, variables, operators, control flow, arrays, and objects. This is the bedrock everything else is built on. Don't skip anything; gaps here cause bugs everywhere.

0%
types done ✓
Variables & Data Types
var is function-scoped and hoisted. let/const are block-scoped. Use const by default — it signals intent, not immutability of the value inside an object.
  • var, let, const — scoping rules and when to use each one
  • Primitive types: string, number, boolean, null, undefined, symbol, BigInt
  • Type coercion: implicit vs explicit, truthy/falsy values, == vs ===
  • typeof operator — and the typeof null === "object" gotcha
control flow done ✓
Operators & Control Flow
Short-circuit evaluation: a && b returns b only if a is truthy. a || b returns a if truthy, else b. a ?? b returns b only if a is null or undefined — not just falsy.
  • Arithmetic, comparison, logical operators — operator precedence
  • Ternary operator and short-circuit evaluation (&&, ||, ??) as expressions
  • if/else, switch — exhaustive matching and fallthrough behaviour
  • Loops: for, while, for...of, for...in — and when to use each
data structures done ✓
Arrays & Objects
.map() returns a new array. .filter() returns a new array. .reduce() returns whatever you want. None of them mutate the original. This non-mutation principle is the foundation of functional JS.
  • Array methods: map, filter, reduce, forEach, find, findIndex, some, every, flat
  • Object literals: property shorthand, computed keys, optional chaining (?.) usage
  • Destructuring: array and object — renaming, defaults, nested destructuring
  • Spread (...) and rest (...) — same syntax, opposite purposes
built-ins done ✓
Strings, Numbers & Built-ins
Template literals aren't just nicer string concatenation. Tagged templates let frameworks like styled-components and GraphQL parse them at build time. Understand the base first.
  • String methods: split, join, trim, replace, includes, slice, padStart, at()
  • Template literals, multi-line strings, and tagged template functions concept
  • Math: floor, ceil, round, random, max/min — and Number: isNaN, isFinite, toFixed
  • Map, Set, WeakMap, WeakSet — when objects and arrays aren't the right tool
◆ Practice Project
Vanilla JS Quiz App
Build a 10-question quiz in the browser console — no DOM manipulation yet. Questions stored as arrays of objects. Track score, shuffle questions using Fisher-Yates algorithm, categorise results. Pure JS fundamentals: loops, conditionals, array methods, objects. No frameworks.
Mark project as complete
⟡

Senior tip: Understand type coercion deeply — it's the source of most JS interview questions. Know why [] + {} and {} + [] give different results. Learn the difference between null and undefined in actual data flow, not just definition. These details separate juniors from seniors.

⬡ Mental Model — Section 2
Everything in JavaScript is either a primitive or an object. Primitives are compared by value; objects are compared by reference. This one fact explains why [] === [] is false, why passing an object to a function can mutate the original, and why shallow vs deep clone matters.

03

DOM & Browser Events

This is where JavaScript meets the browser. Selecting, creating, and manipulating elements, listening to events, and using browser APIs. React, Vue, and Svelte all abstract this — but they all build on top of exactly these concepts.

0%
dom core done ✓
DOM Fundamentals
querySelector returns the first match or null. querySelectorAll returns a NodeList — not an Array. Spread it with [...nodeList] to use array methods.
  • DOM tree: document node, element nodes, text nodes, HTMLCollection vs NodeList
  • Selecting: querySelector, querySelectorAll, getElementById, getElementsByClassName
  • Traversing: parentElement, children, firstElementChild, nextElementSibling
  • Reading: innerHTML, textContent, value, getAttribute, dataset (data-* attributes)
events done ✓
Events & Event Handling
event.target = where the event started. event.currentTarget = where the listener is attached. Events bubble up from child → parent → document. That's why delegation works.
  • addEventListener(type, handler, options) — and removeEventListener
  • Event object: target, currentTarget, type, key, clientX/Y, preventDefault()
  • Event bubbling (bottom-up) vs capturing (top-down) — and stopPropagation()
  • Event delegation — one parent listener instead of N child listeners
manipulation done ✓
DOM Manipulation
innerHTML is fast but opens XSS vulnerabilities — never use it with user input. textContent is safe. createElement is verbose but safest for dynamic content.
  • Creating: createElement, createTextNode, cloneNode(deep)
  • Inserting: appendChild, prepend, append, insertAdjacentHTML, before/after
  • Removing: removeChild, element.remove() — and replaceChild / replaceWith
  • classList API: add, remove, toggle, contains, replace — your most-used DOM tool
web apis done ✓
Browser APIs
localStorage survives browser restarts. sessionStorage dies when the tab closes. Always JSON.stringify before storing and JSON.parse after reading — localStorage only stores strings.
  • localStorage and sessionStorage: getItem, setItem, removeItem, clear
  • fetch() basics: GET request, .json(), .text(), response.ok status check
  • Timers: setTimeout, setInterval, clearTimeout, clearInterval, requestAnimationFrame
  • Window: location, history.pushState, scrollTo, innerWidth/Height, matchMedia
◆ Practice Project
Interactive Todo List
Full CRUD todo app using only vanilla DOM manipulation — no frameworks. Add tasks, mark complete, delete, filter by status (all/active/done). Persist to localStorage. Use event delegation for the task list. This exercises nearly every DOM API you've learned.
Mark project as complete
⟡

Senior tip: Event delegation is not just a performance trick — it's the only correct pattern when you're dynamically adding/removing elements. React uses a single root event listener for the entire app. Understanding why this is necessary before learning React makes the framework make sense immediately.

⬡ Mental Model — Section 3
The DOM is not JavaScript — it's a C++ API that JavaScript talks to. Every DOM read or write crosses a boundary between the JS engine and the browser rendering engine. This is why batching DOM changes matters and why frameworks like React exist — virtual DOM batches your mutations.
querySelector Event delegation classList localStorage fetch basics

Functions, Scope & Closures

The deepest, most important section. Closures power React hooks, Redux, middleware, memoization, and module systems. this causes the most confusion in JS. Master both and you'll understand frameworks at their core.

0%
scope done ✓
Scope & Hoisting
JavaScript determines scope at write time (lexical scope), not at call time. Where you define a function — not where you call it — determines which variables it can access. This is the root of closures.
  • Scope chain: global → function → block scope and the lookup process
  • var hoisting: declarations moved to top of function, initializations stay put
  • Temporal Dead Zone (TDZ): let/const hoisted but inaccessible until declaration
  • Lexical scope: child functions see parent scope, but never the reverse
closures done ✓
Closures
A closure is a function + a reference to its surrounding scope at the time it was created. The outer function returns and its execution context is gone — but the closure keeps the scope alive. This is private state in JavaScript.
  • What closures are: function packaged with its surrounding lexical environment
  • Practical uses: counter factories, private variables, partial application
  • Module pattern: encapsulated stateful objects using closure + IIFE
  • Classic var-in-loop gotcha — and why let fixes it (new binding per iteration)
this keyword done ✓
this & Binding Rules
4 binding rules in priority order: new > explicit (call/apply/bind) > implicit (obj.method()) > default (global/undefined in strict). Arrow functions skip all rules — they inherit this from the enclosing lexical scope.
  • Default binding: global in sloppy, undefined in strict mode
  • Implicit binding: obj.method() — and how storing a method loses `this`
  • Explicit binding: call(ctx, args), apply(ctx, [args]), bind(ctx) — differences
  • Arrow functions: no own `this` — captures lexical `this` from definition site
composition done ✓
Higher-Order Functions
Every time you pass a function to setTimeout, addEventListener, or array methods — you're doing higher-order programming. Internalise this and React hooks, Redux middleware, and Express middleware all feel natural.
  • First-class functions: assigned to vars, passed as args, returned from functions
  • Implementing map, filter, reduce from scratch using a for loop
  • Function composition: compose(f, g)(x) === f(g(x)) — pipe is reversed
  • Currying: f(a, b, c) → f(a)(b)(c) — and when to actually use it
◆ Practice Project
Utility Function Library
Build a mini utility library from scratch — implement debounce, throttle, memoize, deepClone, compose, and once (function that can only be called once). No lodash. Write tests for each one using console.assert. Every function exercises closures directly.
Mark project as complete
⟡

Senior tip: Closures are the source of memory leaks when you close over large DOM trees or datasets. If a closure holds a reference to a DOM node that gets removed, the node cannot be garbage-collected. Always null out references in cleanup functions. This is exactly what React's useEffect cleanup does.

⬡ Mental Model — Section 4
A closure is a backpack attached to a function. Wherever you carry the function, the backpack goes. Inside the backpack are all the variables from the scope where the function was defined — not where it was called. React's useState hook is a closure. Every hook is a closure.
Closures Lexical scope this binding HOF Currying Debounce
05

Async JavaScript

The event loop is the most important mental model in JS. Promises and async/await are just tools built on top of it. Understand the loop first — everything else becomes obvious.

0%
event loop done ✓
The JavaScript Event Loop
JavaScript is single-threaded — one call stack, one thing at a time. The event loop picks tasks from queues when the stack is empty. Microtasks (Promises) always drain before the next macrotask (setTimeout). This ordering is everything.
  • Single-threaded model: one call stack, one execution context at a time
  • Web APIs: where setTimeout, fetch, addEventListener actually live (not in V8)
  • Microtask queue (Promises, queueMicrotask) vs macrotask queue (setTimeout)
  • Classic question: log order of setTimeout(fn, 0) vs Promise.resolve().then(fn)
promises done ✓
Callbacks & Promises
A Promise is a box that will eventually contain a value. You pass the box around synchronously. Only .then() schedules what to do when the box opens. The box is immutable once settled — you can't re-resolve it.
  • Callbacks: the original pattern — and why nesting them creates "callback hell"
  • new Promise((resolve, reject)) — creating and settling promises
  • .then(), .catch(), .finally() — chaining and error propagation through the chain
  • Promise states: pending → fulfilled or rejected — state is immutable once settled
async / await done ✓
Async / Await
async always returns a Promise, even if you return a plain value — it's wrapped automatically. await pauses the async function, not the entire program.
  • async function: always returns a Promise, even for non-Promise return values
  • await: pauses the async function until the Promise settles
  • Error handling: try/catch/finally inside async functions — and re-throwing
  • Sequential vs parallel: await in sequence vs Promise.all([a(), b(), c()])
advanced patterns done ✓
Advanced Async Patterns
Promise.all fails fast — one rejection kills it. Promise.allSettled waits for all and gives you each result. Use allSettled when you need all results regardless of individual failures (e.g. batch API calls).
  • Promise.all (fail fast), Promise.allSettled, Promise.race, Promise.any
  • AbortController: cancelling fetch requests and async operations mid-flight
  • Debounce vs throttle — implement both without libraries
  • Retry logic with exponential backoff — a production async pattern
◆ Practice Project
GitHub Profile Viewer
Fetch the GitHub REST API to display user profiles and repos. Implement async/await with try/catch error handling, AbortController to cancel stale requests on new searches, debounced search input, and a loading skeleton. Covers the entire async chapter in one project.
Mark project as complete
⟡

Senior tip: The most common async mistake is calling await in a loop sequentially when you could use Promise.all. for (const item of items) { await fetch(item) } is N requests in series. await Promise.all(items.map(fetch)) is N requests in parallel — potentially N× faster.

⬡ Mental Model — Section 5
The call stack is a stack of plates. JavaScript adds a plate when a function is called, removes it when the function returns. The event loop only adds new plates when the stack is empty. async/await pauses a function (removes its plate temporarily) without blocking — the loop keeps running other tasks.
Event loop Promises async/await Promise.all AbortController

OOP & Design Patterns

JavaScript's OOP is prototype-based, not class-based. ES6 class syntax is sugar over prototypes. Understanding the prototype chain explains how all built-in methods work — and demystifies every JavaScript framework.

0%
prototypes done ✓
Prototypes & Prototype Chain
When you access a property, JS walks the chain: own properties → prototype → prototype's prototype → null. Array.prototype.map lives on the prototype — not copied onto every array. This is how built-ins work.
  • Every object has [[Prototype]] — accessed via Object.getPrototypeOf()
  • Prototype chain lookup: own properties first, then the chain, then null
  • Object.create(proto): create an object with a specific prototype object
  • hasOwnProperty vs in operator — own properties vs inherited ones
classes done ✓
ES6 Classes
ES6 class is syntactic sugar over prototypes — there are no real classes in JavaScript. The prototype chain still runs underneath. Learn prototypes first; classes second. Classes make the familiar pattern easier to write.
  • class, constructor, instance methods — and how they link to the prototype
  • Getters and setters: get/set with computed property behaviour
  • static methods and properties: called on the class, not on instances
  • Private class fields with #: true encapsulation, enforced by the engine
inheritance done ✓
Inheritance & Composition
Prefer composition over inheritance. Deep inheritance trees are fragile — changing a base class breaks every subclass below it. Compose behaviour through mixins and higher-order functions instead of extending.
  • extends keyword and how it sets up the prototype chain between classes
  • super() in constructor — must call before accessing `this` in subclass
  • Method overriding and the fragile base class problem
  • Mixin pattern: Object.assign(Target.prototype, Mixin) for composition
patterns done ✓
Design Patterns
Patterns are vocabulary — they communicate complex ideas in one word. Observer = subscribe/publish. Factory = create without new. Module = encapsulate. Singleton = one instance. Know the pattern, know its tradeoffs.
  • Module pattern (IIFE + closure): encapsulation before ES modules existed
  • Factory functions: create objects without the new keyword — more flexible
  • Observer / Pub-Sub: subscribe(), publish(), unsubscribe() — build it from scratch
  • Singleton, Strategy, and Decorator patterns — and their real-world JS usage
◆ Practice Project
OOP Bank Account System
Build Account, SavingsAccount, and CheckingAccount classes using inheritance. Private fields for balance. Static factory methods for creating accounts. Observer pattern to notify "transaction listeners" on every deposit/withdraw. Compose an ATM class that uses dependency injection instead of extending Account.
Mark project as complete
⟡

Senior tip: React components are functions (or classes) that follow the Factory pattern. React hooks are closures. Context is Observer pattern. Redux is a variant of Flux which is a strict Observer. Once you see patterns everywhere, you stop needing to memorise framework-specific APIs — you already understand the shape.

⬡ Mental Model — Section 6
The prototype chain is a linked list of objects. When you do arr.map(), JavaScript doesn't find map on the array — it finds it on Array.prototype. Every array in the program shares one copy of map. Classes don't change this — they just make writing prototype links look familiar to Java/Python developers.
Prototype chain ES6 Classes extends / super Observer Factory
07

Advanced JavaScript

Iterators, generators, Proxy, memory management, and modern ES features. This is what separates engineers who use JavaScript from engineers who understand it. You'll see how frameworks are built, not just how to use them.

0%
iteration protocol done ✓
Iterators & Generators
for...of works on anything with a Symbol.iterator — arrays, strings, Maps, Sets, generators. This protocol lets you make your own data structures work with for...of and spread.
  • Iterable protocol: Symbol.iterator, the iterator object, .next() → {value, done}
  • for...of under the hood — how it calls Symbol.iterator and .next()
  • Generator functions: function*, yield, pausing and resuming execution
  • Infinite sequences and lazy evaluation — generating values on demand
meta-programming done ✓
Proxy & Reflect
Proxy wraps an object and intercepts operations on it. Vue 3's entire reactivity system is built with Proxy — when you set a property, the Proxy trap fires and triggers component re-renders. This is not a niche feature.
  • new Proxy(target, handler): intercepting get, set, has, deleteProperty, apply
  • Reflect API: the counterpart to every Proxy trap — consistent meta-operations
  • Building a reactive object: notify watchers on set, track dependencies on get
  • Use cases: validation, auto-logging, read-only views, virtual properties
performance done ✓
Memory & Performance
Memory leaks happen when you keep references to objects you no longer need. Most common sources: event listeners never removed, global variables, closures holding DOM references. WeakMap breaks these reference cycles.
  • Garbage collection: mark-and-sweep — how GC determines what to collect
  • Common leak patterns: detached DOM nodes, unremoved listeners, accidental globals
  • WeakMap and WeakSet: hold weak references that don't prevent GC
  • DevTools: Memory tab (heap snapshots), Performance tab, flame charts
modern features done ✓
Modules & Modern ES Features
ES Modules are live bindings — when the exported value changes, all importers see the new value. CommonJS copies the value at require() time. This difference causes bugs in large codebases. Know which module system you're using.
  • ES Modules: named export/import, default export/import, re-exporting with as
  • CommonJS vs ESM: require() vs import — sync vs async, copies vs live bindings
  • Optional chaining (?.), nullish coalescing (??), logical assignment (&&=, ||=, ??=)
  • structuredClone, Array.at(), Object.fromEntries, Error.cause, top-level await
◆ Capstone Project

Add a computed(fn) function for derived values. Add a watch(fn) for side effects. This is the core of Vue, MobX, and Solid.js.

⟡

Senior tip: After completing this section, read the Vue 3 reactivity source code. It's Proxy-based and remarkably readable. Then read the Solid.js source. You'll recognise every pattern — because you just built them. Reading framework source is how engineers go from senior to staff level.

⬡ Mental Model — Section 7
Generators pause execution and resume it — they're cooperative multitasking inside a single thread. The async/await syntax was built on generators. Proxy intercepts the language itself. These aren't "advanced features" — they're the implementation substrate that all modern JavaScript tooling is built on top of.
